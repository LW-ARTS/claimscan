# Phase 12.1 Plan 08: Production Migration + Reclassification + Validation

Final wave of Phase 12.1 — applied migration to live DB, ran classify-flap.ts loop until queue drained, validated split-vault count via MCP SQL.

## Task 1: Apply migration 035 to PRODUCTION

Combined with Plan 02 staging apply (single project — no staging/prod separation in this codebase). Used Supabase MCP `apply_migration` tool against project `qjbqsavyfsfanutlediy`. Migration applied successfully, recorded as `035_add_splitvault_to_check` in `supabase_migrations.schema_migrations`.

**Verification (post-apply):**

```sql
SELECT conname, pg_get_constraintdef(oid) FROM pg_constraint
WHERE conname IN ('flap_tokens_vault_type_check', 'fee_records_vault_type_check');
```

Both predicates now include `'split-vault'`. Canary INSERT/DELETE round-trip with `vault_type='split-vault'` succeeded.

## Task 2: Run classify-flap.ts (loop until queue drained)

Pre-classify state (after migration applied):
- unknown: 6720
- base-v2: 72
- base-v1: 0
- split-vault: 0 (constraint accepts but no handler had populated it yet)

The Supabase JS client caps queries at 1000 rows by default, so a single `SAMPLE_LIMIT=10000` invocation still only processed 1000 rows per run. Solved with a bash loop (`/tmp/classify-flap-12.1-loop.sh`) that runs classify-flap.ts repeatedly until the script reports "Nothing to classify" (queue drained).

Initial batch (run before loop):
- 1000 rows processed: 641 split-vault, 310 base-v2, 49 unknown-failed
- Hit rate: 64.1% split-vault (matches research's 58.6% within sample variance)

Loop iterations: 10 ASC + parallel DESC (DESC killed after ASC drained, both running concurrently halved wallclock from ~85 min to ~70 min)
Total rows processed in loop: ~5479 unknowns drained to split-vault/base-v2/unknown-terminal classifications
Loop duration: ~70 min wallclock (ASC parallel with DESC, both targeting opposite ends of the queue via SAMPLE_ORDER_DESC=1)
Halt condition: ASC iteration 10 ran 313 rows that all returned 'unknown' (probes terminal — these are vault impls outside V1/V2/SplitVault patterns). DESC was killed at this point since further iterations would just re-process the same 313 stuck-unknowns.

## Task 3: DB Sanity SELECT (validation)

Final query via MCP `execute_sql`:

```sql
SELECT vault_type, COUNT(*) AS cnt
FROM flap_tokens
WHERE vault_address IS NOT NULL
  AND vault_address != '0x0000000000000000000000000000000000000000'
GROUP BY vault_type
ORDER BY cnt DESC;
```

Final state:

| vault_type | count | % of vault-having tokens |
|------------|-------|--------------------------|
| split-vault | **4001** | 58.9% |
| base-v2 | 2478 | 36.5% |
| unknown | 313 | 4.6% (terminal — probes failed for unrecognized vault impls, Phase 12 D-04 badge handles these) |
| **Total** | **6792** | 100% (matches initial unknowns + base-v2 from research) |

**Acceptance criterion:** `split-vault` count in range `[3500, 4500]` (per research §"Population & Steady-State Behavior"). Result: **4001 ✓**

Research projected 3979 SplitVault instances (58.6%). Actual: 4001 (58.9%). Within +0.3 pp of projection — research empirically validated post-deploy.

The 2478 base-v2 count is much higher than the original 72. The loop reclassified ~2406 rows that Phase 12's classify-flap had previously marked 'unknown' (likely due to timing/RPC issues on first deploy). Phase 12.1's loop correctly classified them as base-v2.

## Threats Mitigated

- **T-12.1-MIG:** Existing 'unknown' rows that probe-out to 'split-vault' need migration BEFORE update — applied migration first via MCP, then ran classify-flap. Verified via canary INSERT before queue processing.
- **T-12.1-CHK:** Both `flap_tokens_vault_type_check` AND `fee_records_vault_type_check` updated atomically in same migration. Pitfall 1 mitigation. Verified via `pg_get_constraintdef`.
- **T-12.1-DRIFT:** Probe ladder mirror in scripts/classify-flap.ts (Plan 06) ensures script and library agree on classification. Production runs use the script (cron disabled), library used by app runtime.

## Steady-State Behavior Note

Per research, most SplitVault clones auto-dispatch on each trade — `accumulated == claimed` is the steady state. After reclassification, app-side `flapAdapter.getHistoricalFees()` will filter zero-claimable rows (per Phase 12 D-12). Net visible UI change: many Flap creators will see their tab disappear or get quieter compared to Phase 12 (where they saw misleading "Claim method unknown" badges). This is correct behavior, not a regression.
