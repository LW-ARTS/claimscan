# Phase 12.1 Plan 02: Migration 035 — Widen vault_type CHECK Constraints

Extended both `flap_tokens.vault_type` and `fee_records.vault_type` CHECK constraints to accept the new `'split-vault'` value, so the upcoming SplitVault handler can persist its classification without violating runtime invariants.

## What Was Built

Two SQL files committed to `supabase/migrations/`:
- `035_add_splitvault_to_check.sql` — DROP+ADD pair on both tables in a single BEGIN/COMMIT transaction
- `035_add_splitvault_to_check_rollback.sql` — companion revert with WARNING note about pre-cleaning split-vault rows

Migration applied to production Supabase project `qjbqsavyfsfanutlediy` via `mcp__plugin_supabase_supabase__apply_migration`.

## Verification (post-apply)

```sql
SELECT conname, pg_get_constraintdef(oid) FROM pg_constraint
WHERE conname IN ('flap_tokens_vault_type_check', 'fee_records_vault_type_check');
```

Returned 2 rows, both predicates now include `'split-vault'`:
- `flap_tokens_vault_type_check`: `CHECK ((vault_type = ANY (ARRAY['base-v1'::text, 'base-v2'::text, 'split-vault'::text, 'unknown'::text])))`
- `fee_records_vault_type_check`: `CHECK (((vault_type IS NULL) OR (vault_type = ANY (ARRAY['base-v1'::text, 'base-v2'::text, 'split-vault'::text, 'unknown'::text]))))`

Canary INSERT/DELETE round-trip with `vault_type='split-vault'` succeeded → status `canary_passed`.

Pre-apply state: 6720 unknown + 72 base-v2 + 0 base-v1 = 6792 vault tokens (matches RESEARCH §"Population & Steady-State Behavior").
